# i18n and translation rules

## Key structure

- Store i18n dictionaries under `ui.apps/src/main/content/jcr_root/apps/<project>/i18n/`
- One JSON or XML file per language — filename must be the IETF language tag (e.g. `en.json`, `de.json`, `fr_FR.json`)
- All languages must contain the same set of keys — missing keys silently fall back to the key string itself

## Key naming

- Use dot-separated namespacing: `product.card.title`, `nav.search.placeholder`
- Keys are case-sensitive — `product.Title` and `product.title` are different keys
- Never use the translated string as a key — keys must be stable identifiers
- Do not include HTML in translation values — translate text only, apply markup in HTL

## HTL usage

- Use `${i18n.get('key')}` or `${i18n.get('key', 'comment')}` — the comment helps translators understand context
- Apply `@ context='text'` to all i18n expressions: `${i18n.get('key') @ context='text'}`
- Never use `context='unsafe'` for translated strings — translator-supplied content is untrusted

## Java usage

- Inject `I18n` via `ResourceBundle` from `SlingHttpServletRequest.getResourceBundle(locale)`
- Do not hardcode locale — derive it from the page or request
- Use `i18n.get(key)` — never concatenate translated strings with other strings

## Review focus

- Does every language file contain all keys present in the default (`en`) file?
- Are i18n expressions in HTL using `@ context='text'`?
- Are keys stable identifiers, not human-readable strings?
- Is the locale derived from the request or page context, not hardcoded?
- Are translation files placed in `ui.apps` under the correct `i18n/` path?
